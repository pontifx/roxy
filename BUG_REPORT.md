## Bug 1: Envoy `path_prefix` Causes All Buyer Proxy Requests to Receive 404/Denial

**File:** `envoy/envoy.yaml` and `ext-authz/src/server.js`  
**Severity:** Critical  
**Description:**  
In Envoy's HTTP ext_authz filter, the `path_prefix` field prepends the specified prefix to the **original request path** before forwarding to the authorization service. With `path_prefix: /check`, if a buyer makes a request to any path (e.g., `/products/123`), Envoy calls `POST /check/products/123` on ext-authz — not `POST /check`.

The ext-authz server only registers `app.post('/check', ...)`, which matches **only the exact path** `/check`. Express does not match `/check/products/123` to that route, so it falls through to a 404 response. Since `failure_mode_allow: false` is set in envoy.yaml, any non-2xx response from ext-authz causes Envoy to **deny the request**. This means every buyer proxy request is denied with a confusing, silent 404 from the authz service rather than being authorized.

The developer's comment in envoy.yaml reads: "The path on the authz service that Envoy will POST to", which shows the intended behavior was an exact endpoint, not a prefix prepend. The fix must be applied to ext-authz to accept any path under `/check/*`.

**Fix:**  
Changed `app.post('/check', ...)` to `app.post('*', ...)` in ext-authz so it accepts the check request regardless of what path Envoy appends. The handler already extracts authorization data from request headers (not from the path), so the path change has no semantic effect. The `path_prefix: /check` in envoy.yaml is intentionally preserved as a clean namespace separator.

## Bug 2: `fetch` Race Condition in ext-authz — Undefined on First Requests

**File:** `ext-authz/src/server.js`  
**Severity:** High  
**Description:**  
`node-fetch` v3 is ESM-only. The server works around this in a CommonJS file by assigning it via an async IIFE:

```js
let fetch;
(async () => { fetch = (await import('node-fetch')).default; })();
```

`fetch` is `undefined` until the promise resolves. The HTTP server starts immediately (`app.listen`) and the `POST /check` handler references `fetch` synchronously. Any request that arrives in the milliseconds between server start and the IIFE resolving will call `fetch(...)` where `fetch` is `undefined`, throwing `TypeError: fetch is not a function`. Under Envoy's `failure_mode_allow: false`, this causes an internal error denial for the first few requests — or worse, crashes the authz service.

**Fix:**  
Server startup is deferred until `fetch` is initialized. The IIFE is replaced with a top-level async `start()` function that awaits the `node-fetch` import before calling `app.listen(...)`. This guarantees `fetch` is defined before any request can be handled.

## Bug 3: `module.exports.TIERS` Silently Lost in `packages.js`

**File:** `session-manager/src/routes/packages.js`  
**Severity:** High  
**Description:**  
The file has two conflicting export statements:

```js
// Line 42
module.exports.TIERS = TIERS;   // Sets a property on the initial exports object

// Line 173 (much later, after all route definitions)
module.exports = router;         // REPLACES the entire exports object
```

When Node.js executes this file, the final `module.exports = router` completely replaces the exports object, discarding the `TIERS` property set earlier. Any code that does `require('./routes/packages').TIERS` will receive `undefined`. The in-file comment "Export TIERS so other modules (packageManager, API docs) can reference them" documents the intent but the implementation is broken.

**Fix:**  
Removed the dead `module.exports.TIERS = TIERS` line at the top. Added `router.TIERS = TIERS` before `module.exports = router` at the bottom, which correctly attaches `TIERS` as a property of the exported router. Consumers can access it as `require('./routes/packages').TIERS`.

## Bug 4: `findAvailableSession` Imported but Never Used in `sessions.js`

**File:** `session-manager/src/routes/sessions.js`  
**Severity:** Medium  
**Description:**  
Line 25 imports `findAvailableSession` from `../services/sessionPool`:

```js
const { findAvailableSession, claimSession, releaseSession } = require('../services/sessionPool');
```

`findAvailableSession` is never called in this file. The `/claim` route delegates to `claimSession()`, which internally calls `findAvailableSession`. This creates a misleading dead import that clutters the code and confuses future maintainers about whether `findAvailableSession` is meant to be called directly from routes.

**Fix:**  
Removed `findAvailableSession` from the destructured import.

## Bug 5: `webRequestAuthProvider` is an Unused and Incorrect Manifest Permission

**File:** `extension/manifest.json`  
**Severity:** Medium  
**Description:**  
The manifest declares `"webRequestAuthProvider"` in its `permissions` array. This permission is associated with the `chrome.webAuthenticationProxy` API (a completely different, enterprise-focused API for intercepting WebAuthn/FIDO2 authentication ceremonies). The Roxy extension never uses `chrome.webAuthenticationProxy` anywhere in its codebase.

This is not the permission for handling HTTP authentication challenges (which would use `webRequest` and `onAuthRequired`). Having this permission in the manifest requests capabilities the extension does not need, which:
1. Widens the attack surface unnecessarily.
2. May trigger Chrome Web Store security reviews.
3. Requires user consent for an enterprise-level permission the extension doesn't use.

**Fix:**  
Removed `"webRequestAuthProvider"` from the `permissions` array.

## Bug 6: Unused `findAvailableSession` Export in `sessions.js` Destructure (see Bug 4) — Additional: `purchased_at` Field Silently Ignored in SQL INSERT

**File:** `session-manager/src/services/packageManager.js` and `session-manager/src/db/store.js`  
**Severity:** Low  
**Description:**  
`packageManager.js` builds a `pkg` object that includes `purchased_at: new Date().toISOString()` and passes it to `dbCreatePackage(pkg)`. However, `store.js` `createPackage` SQL does not include `purchased_at` in its `INSERT` column list:

```sql
INSERT INTO packages (id, buyer_id, tier, sessions_total, sessions_remaining,
                      req_res_total, req_res_remaining, expires_at)
VALUES (@id, @buyer_id, @tier, ...)
```

`better-sqlite3` named parameter binding ignores extra keys not in the column list, so `purchased_at` from the caller is silently discarded. SQLite uses the column default `datetime('now')` instead. The immediate `pkg` object returned by `createPackage` in packageManager.js contains the JavaScript `purchased_at` value, while the SQLite record contains the SQL-generated value — these can differ by a few microseconds. Any subsequent `getPackage()` call returns the SQLite value, creating a subtle inconsistency in the returned data.

**Fix:**  
Added `purchased_at` to the `INSERT` column list in `store.js` `createPackage`. This ensures the caller-supplied ISO timestamp is stored exactly, making the in-memory returned object consistent with the persisted record.

## Items Confirmed Correct (No Bug)

- **Syntax:** All `.js` files pass `node --check` with no syntax errors.
- **Docker networking:** Service names in `docker-compose.yml` (`envoy`, `session-manager`, `ext-authz`, `redis`) exactly match the hostnames used in `envoy.yaml` clusters and in code (`http://session-manager:3000`, `ext-authz:9191`, `redis://redis:6379`).
- **WebSocket path:** `hostConnection.js` listens on `/ws/host`, `envoy.yaml` routes `/ws/*` prefix to `session_manager_cluster` (bypassing ext_authz), and `background.js`/`popup.js`/`options.js` all default to `wss://localhost:10000/ws/host`. All consistent.
- **JWT flow:** `signToken` in `auth.js` and `verifyToken` in `middleware/auth.js` use the same `JWT_SECRET`/`JWT_TTL` env vars. `internal.js` uses `verifyToken` and correctly reads `decoded.id` and `decoded.role`. Token payload fields (`id`, `email`, `role`) are consistent across sign and verify.
- **DB schema vs queries:** All column names in SQL queries (`session_id`, `buyer_id`, `host_id`, `host_share`, `buyer_share`, `proof_url`, `created_at`, etc.) match the `CREATE TABLE` definitions in `initDatabase()`.
- **Internal route path:** `ext-authz` calls `http://session-manager:3000/api/internal/validate`; `index.js` mounts `app.use('/api/internal', internalRoutes)` and `internal.js` defines `router.post('/validate', ...)`. Path matches.
- **`internalOnly` guard:** The `X-Internal-Service` header sent by `ext-authz` matches the check in `internal.js`.
- **MV3 service worker background:** `manifest.json` correctly uses `"type": "module"` for `background.js`; `background.js` does not use `'use strict'` (correct — ES modules are implicitly strict).
- **Keepalive alarm:** `chrome.alarms.create('roxy_keepalive', { periodInMinutes: 0.4 })` runs every ~24s, under the 30s MV3 service worker idle timeout.
- **`atomicClaimSession` Redis WATCH:** Correctly uses a duplicate client for WATCH/MULTI to avoid interfering with the main client pipeline.
- **Route ordering:** No GET/DELETE method conflicts in `sessions.js`. Static routes (`/available`, `/mine`) are on GET; `/:id` is on DELETE — no Express routing collision.
- **Envoy cluster names:** `ext_authz_cluster` referenced in `server_uri.cluster` matches the cluster name defined in `static_resources.clusters`. `session_manager_cluster` referenced in route configs matches the cluster definition.
